# การปรับโครงสร้าง Workflow ตามลำดับชั้นที่ถูกต้อง

อ้างอิงจาก Feedback ของคุณวันเฉลิม ผมเข้าใจโครงสร้างและลำดับชั้นของ กทม. อย่างชัดเจนแล้วครับ โครงสร้างที่ถูกต้องคือ:
- `HR_DIRECTOR` และ `DIV_DIRECTOR` อยู่ระดับเดียวกัน (สูงสุด)
- `SEC_DIRECTOR` มีหรือไม่มีก็ได้
- `GRP_LEADER`
- `STAFF`
- `COORDINATOR` 

และเส้นทางการเดินเอกสารจะเป็นการ **วิ่งขึ้น (เสนอ)** ไปจนถึงระดับผู้อำนวยการ จากนั้น **วิ่งข้ามสายงาน** แล้ว **วิ่งลง (มอบหมาย)** ไปจนถึงระดับปฏิบัติการ (STAFF) แล้ว **วิ่งกลับขึ้นมาใหม่** จนถึงผู้อำนวยการ แล้ว **ส่งกลับ** มายังคนตั้งเรื่อง (COORDINATOR)

เพื่อให้ระบบสามารถรองรับเส้นทางที่ซับซ้อนและยืดหยุ่น (กรณีที่บางกองไม่มี SEC_DIRECTOR) โดยที่ Status ไม่รวน ผมขอเสนอแผนการรื้อลอจิกการเปลี่ยนสถานะ (Status Routing) ใหม่ทั้งหมด ดังนี้ครับ:

## 1. การเพิ่ม Status ให้ครบตามบริบท
ผมจะเพิ่ม Status ในฐานข้อมูลและในโค้ดให้ครบตามที่คุณวันเฉลิมระบุเป๊ะๆ:
- `DRAFT`
- `PENDING_GRP_REVIEW`
- `PENDING_SEC_APPROVAL`
- `PENDING_DIRECTOR_APPROVAL`
- `PENDING_HR_APPROVAL`
- `PENDING_DELEGATION`
- `PENDING_EXECUTION`
- `PENDING_CLOSE`
- `COMPLETED`
- `REJECTED`

## 2. การใช้ระบบ "คำนวณ Status อัตโนมัติด้วย Rank (ระดับชั้น)"
แทนที่เราจะเขียนโค้ดผูกมัดว่าปุ่มไหนเปลี่ยนเป็น Status อะไร (ซึ่งเสี่ยงต่อบั๊กมาก) ผมจะสร้างระบบประเมินทิศทางของเอกสารอัตโนมัติ โดยให้คะแนน Rank กับทุก Role เช่น:
- STAFF = 1
- GRP_LEADER = 2
- SEC_DIRECTOR = 3
- DIV_DIRECTOR / HR_DIRECTOR = 4
- COORDINATOR = 0 (กรณีพิเศษ)

**เมื่อมีการส่งเอกสาร (Forward) ระบบจะเปรียบเทียบ Rank ของคนส่ง กับคนรับ:**
1. **ถ้าส่งไปหา COORDINATOR:** สถานะจะเป็น `PENDING_CLOSE` ทันที
2. **ถ้าส่งไปหา STAFF:** สถานะจะเป็น `PENDING_EXECUTION` ทันที
3. **ถ้า Rank คนส่ง > Rank คนรับ (ทิศทางวิ่งลง):** แสดงว่าเป็นการ "มอบหมายงานลงมา" สถานะจะเป็น `PENDING_DELEGATION` โดยอัตโนมัติ 
   *(เช่น DIV_DIRECTOR ส่งให้ SEC_DIRECTOR หรือ SEC_DIRECTOR ส่งให้ GRP_LEADER จะได้สถานะนี้)*
4. **ถ้า Rank คนส่ง <= Rank คนรับ (ทิศทางวิ่งขึ้น หรือ วิ่งระนาบเดียวกัน):** แสดงว่าเป็นการ "เสนอเรื่อง" หรือ "ส่งข้ามหน่วยงาน" ระบบจะตั้งชื่อสถานะตาม **ตำแหน่งของคนรับ** ทันที:
   - รับโดย GRP_LEADER -> `PENDING_GRP_REVIEW`
   - รับโดย SEC_DIRECTOR -> `PENDING_SEC_APPROVAL`
   - รับโดย DIV_DIRECTOR -> `PENDING_DIRECTOR_APPROVAL`
   - รับโดย HR_DIRECTOR -> `PENDING_HR_APPROVAL`

**ข้อดีของระบบนี้:**
- รองรับ Flow ของคุณวันเฉลิมได้ **100% เต็ม** โดยไม่ต้องเขียนโค้ดแยกแต่ละเคสเลยครับ
- ไม่ว่าตรงกลางจะข้าม SEC_DIRECTOR ไปหรือไม่ ระบบก็จะคำนวณ Status ได้ถูกต้องเสมอ
- ทำให้เมนูต่างๆ ใช้งานง่ายขึ้น ผู้ใช้เพียงแค่ค้นหาชื่อคนรับและกด "ส่งต่อ" ระบบหลังบ้านจะรู้เองว่าควรปรับเป็น Status อะไร

## 3. การยุบรวมเมนูและปุ่ม (Simplify UI)
ในหน้าจอ `WorkflowActionModal` จะไม่มีปุ่มที่แยกย่อยเกินไปแล้ว (เช่น ซ่อนปุ่ม ส่งให้ HR, ซ่อนปุ่มส่งให้ GRP_LEADER) แต่จะรวมเป็นปุ่ม **"เสนอเรื่อง / ส่งต่อ"** แทน 
เมื่อเลือกคนรับเสร็จ ระบบหลังบ้าน (จากข้อ 2) จะเป็นคนจัดการให้ทั้งหมด ลดโอกาสการกดปุ่มผิดของผู้ใช้ครับ

## 4. กรณีการ "ตีกลับ" (Reject)
การตีกลับจะใช้สถานะ `REJECTED` ตรงๆ เลย เพื่อให้กล่องข้อความแสดงไอคอนสีแดงชัดเจนว่าเป็นการตีกลับมาให้แก้ไข ไม่ใช่การมอบหมายงานปกติครับ

> [!IMPORTANT]
> **การขอความเห็น:** 
> ระบบคำนวณทิศทางขึ้น-ลงแบบนี้ ตรงกับลอจิกการทำงานของเอกสารราชการในหน่วยงานหรือไม่ครับ? ถ้ายืนยันตามนี้ รบกวนกด Approve ได้เลยครับ ผมจะเข้าไปปรับแก้โครงสร้าง Database Types และลอจิกของ Backend ให้รองรับระบบใหม่นี้ทันทีครับ
