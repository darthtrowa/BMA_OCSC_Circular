# การออกแบบสถาปัตยกรรมกระบวนการทำงานแบบย่อยและไดนามิก (Modular & Dynamic Workflow Design)

เอกสารฉบับนี้สรุปหลักการวิเคราะห์ แนวทางปฏิบัติ และการออกแบบระบบการเดินเอกสาร (Workflow Engine) สำหรับโครงการ **BMA OCSC Circular** เพื่อตอบคำถามเชิงสถาปัตยกรรมว่าควรวางระบบกระบวนการทำงานในรูปแบบกระบวนการย่อยหลายตัว (Modular Workflows) หรือกระบวนการเดียวขนาดใหญ่ (Monolithic Workflow) รวมถึงวิธีการเชื่อมโยงและการส่งต่อข้อมูลภายในองค์กรอย่างมีประสิทธิภาพ

---

## 1. การวิเคราะห์เปรียบเทียบเชิงสถาปัตยกรรม (Architectural Comparison)

ในการพัฒนาระบบเดินเอกสารราชการ มีแนวทางการออกแบบกระบวนการทำงาน 2 รูปแบบหลัก:

### แนวทางที่ 1: แยกกระบวนการทำงานเป็นโมดูลย่อย (Modular Workflows) - *[แนวทางที่เลือกใช้]*
เป็นการแยก Template ของกระบวนการทำงานออกจากกันตามวัตถุประสงค์ เช่น:
*   **Workflow 1:** กระบวนการเสนออนุมัติหลัก (เสนอผ่าน `HR_Director` เพื่ออนุมัติและส่งมอบต่อให้ `DIV_Director`)
*   **Workflow 2:** กระบวนการสั่งการภายในกอง (รับจาก `DIV_Director` มอบหมายลงไประดับ `Staff` ปฏิบัติงาน แล้วเสนอกลับขึ้นมาตามลำดับขั้นเพื่อแจ้งผลให้ `COORDINATOR` รับทราบ)

#### 🟩 ข้อดี:
1.  **การบำรุงรักษาง่าย (High Maintainability):** โครงสร้างบอร์ดในหน้าจอ Workflow Builder (React Flow Canvas) ไม่ซับซ้อน เส้นเชื่อมไม่อีนุงตุงนัง การแก้ไขตัวแปรหรือลำดับขั้นของฝ่ายหนึ่งไม่กระทบกับกระบวนการทำงานของฝ่ายอื่น
2.  **ประสิทธิภาพสูง (Engine Performance):** การประมวลผลเปลี่ยนผ่านสถานะ (State Transition) ของ Backend ไม่ซับซ้อน เพราะไม่มีทางแยกของเงื่อนไขที่ซ้อนกันมากเกินไป
3.  **ประสบการณ์ผู้ใช้ดีเยี่ยม (UX/UI):** ผู้ลากเส้นแก้ไขกระบวนการไม่สับสนกับกิ่งทางเลือกที่หลากหลาย และผู้ใช้ทั่วไปเห็นขั้นตอนการเดินงานเฉพาะส่วนที่เกี่ยวข้องกับตนเอง
4.  **ง่ายต่อการทดสอบ (Unit Testing):** สามารถจำลองการทำงานและเขียน Test Case แยกทีละส่วนได้อย่างสมบูรณ์

#### 🟥 ข้อเสีย:
*   มีจำนวน Template ในระบบเพิ่มขึ้นเล็กน้อย แต่สามารถบริหารจัดการได้ด้วยการจัดหมวดหมู่ (Category Mapping)

---

### แนวทางที่ 2: รวมศูนย์เป็นกระบวนการเดียวยาวตลอดสาย (Monolithic Workflow)
การสร้างผังกระบวนการทำงานขนาดใหญ่เพียงตัวเดียวหรือสองตัว แล้วใช้เงื่อนไข (Condition) บนเส้นเชื่อมเพื่อเปลี่ยนทิศทางการเดินงาน

#### 🟩 ข้อดี:
*   ข้อมูลทุกจุดวิ่งเข้าสู่ผังควบคุมศูนย์กลางชุดเดียวกัน

#### 🟥 ข้อเสีย:
1.  **ความซับซ้อนระดับวิกฤต (Spaghettification):** ผังงานจะมีเส้นเชื่อมลากตัดกันไปมา (Spaghetti Flow) ในหน้าจอ React Flow จะทำให้ผู้ใช้อ่านลำบากมาก
2.  **ความเสี่ยงต่อระบบ (High Risk):** หากระเบียบปฏิบัติหรือโครงสร้างของกลุ่มงานใดกลุ่มงานหนึ่งเปลี่ยน และต้องแก้ไข Node/Edge บนผังงานร่วมนี้ อาจส่งผลกระทบให้เส้นทางการเดินเอกสารอื่น ๆ ในระบบพังทลายลงทั้งหมด (Domino Effect)
3.  **แก้ไขบัคยาก (Debugging Nightmare):** การประมวลผล State Machine เพื่อหาผู้รับงานคนถัดไปต้องคำนวณจากหลายมิติ (Multi-branch Matrix)

---

## 2. วิธีการเชื่อมโยงระบบ Workflow ย่อย (Workflow Linkage Techniques)

เมื่อตัดสินใจแยกกระบวนการอนุมัติหลัก (Workflow 1) และกระบวนการสั่งการ (Workflow 2) ออกจากกัน ระบบสามารถเชื่อมโยงกันได้ผ่าน 3 รูปแบบหลัก:

### รูปแบบ A: การเปลี่ยนสถานะแบบ Event-Driven (Event-Driven State Switch) - *[แนะนำที่สุด]*
*   **หลักการทำงาน:** เมื่อ Workflow 1 ดำเนินการมาถึงจุดสิ้นสุด (เช่น `DIV_Director` ได้รับหนังสือ)
*   **การเชื่อมต่อ:** Backend Engine จะดักจับสถานะ `COMPLETED` ของ Workflow 1 และตรวจสอบกฎของระบบ (System Rule Hooks) เพื่อทำการ:
    1.  เปลี่ยนค่า FK `in_workflow_template_id` ในตาราง `c_information` ไปชี้ที่ **Workflow 2**
    2.  ตั้งค่า `in_current_node_id` ให้เป็น Node เริ่มต้นของ Workflow 2 (ซึ่งก็คือ `DIV_Director` ในฐานะผู้เริ่มส่งต่อ)
    3.  ส่งการแจ้งเตือนไปยัง `DIV_Director` เพื่อให้เข้ามาดำเนินการมอบหมายงานต่อ
*   **เหตุผลที่แนะนำ:** พัฒนาง่าย ไม่กระทบโครงสร้างข้อมูลเดิม มีประวัติการทำงานของเอกสารชุดเดียวกันต่อเนื่องกันชัดเจน

### รูปแบบ B: การเชื่อมโยงผ่าน Action Node (Action-Based Transition)
*   **หลักการทำงาน:** บนเส้น Edge สุดท้ายของ Workflow 1 ที่ชี้ไปหา `DIV_Director` จะมีการฝัง Properties พิเศษในฐานข้อมูล เช่น `action_type = 'START_SUB_WORKFLOW'` และ `target_template_id = [ID_OF_WORKFLOW_2]`
*   **การเชื่อมต่อ:** เมื่อฝั่ง Backend ประมวลผล Action จากคำอนุมัติของ Node สุดท้าย จะนำเอา `target_template_id` นั้นมาล้าง State workflow เดิม และเริ่มใช้งาน Workflow 2 ทันที
*   **เหตุผลที่แนะนำ:** แอดมินสามารถเปิดกว้างในการเลือกเชื่อมโยง Workflow ถัดไปผ่าน UI Builder ได้โดยตรง

### รูปแบบ C: โครงสร้างแม่-ลูก (Parent-Child Workflow)
*   **หลักการทำงาน:** Node ปลายทางใน Workflow 1 ถูกตั้งประเภทให้เป็น "Sub-Workflow Node"
*   **การเชื่อมต่อ:** เมื่อเอกสารเดินมาถึงกล่องนี้ ระบบจะ Freeze สถานะของ Workflow 1 ไว้ แล้วแยกกระบวนการภายในออกไปเป็น Workflow 2 เมื่อ Workflow 2 ดำเนินการจนเสร็จสิ้น (เสนอกลับมาถึง `COORDINATOR` รับทราบผล) จึงจะส่งสัญญาณ Callback กลับมาปลดล็อกและอัปเดตสถานะของ Workflow 1
*   **เหตุผลที่แนะนำ:** เหมาะสำหรับกรณีต้องการวิเคราะห์ประสิทธิภาพการทำงานแบบ Nested (แต่พัฒนาซับซ้อนที่สุด)

## 3. การออกแบบระบบการมอบหมายงานแบบยืดหยุ่น (Ad-hoc Routing)

ในทางปฏิบัติ ผู้อำนวยการกอง (`DIV_Director`) อาจจะมอบหมายงานตรงไปยังผู้อำนวยการส่วน (`SEC_Director`) หรือข้ามไปยังหัวหน้ากลุ่มงาน (`GRP_LEADER`) เลยโดยตรงตามความสำคัญของเอกสาร 

สถาปัตยกรรมที่แนะนำในการออกแบบรองรับกรณีนี้คือ **"Dynamic Target Node (ปลายทางแบบยืดหยุ่น)"** โดยทำงานสอดคล้องกับโครงสร้างองค์กร (Org Chart) ในฐานข้อมูล:

### การตั้งค่าบน Workflow Builder:
1.  ในการลากเส้นผังเชื่อมโยง แอดมินไม่จำเป็นต้องสร้างเส้นเชื่อมแยกเป็น 2 ทางเพื่อจำกัดสิทธิ์
2.  ให้สร้าง Node ถัดจาก `DIV_Director` เพียง Node เดียว โดยระบุประเภทผู้รับผิดชอบ (`assignee_type`) บน Node นั้นให้เป็น **`DYNAMIC_SUBORDINATE`** หรือแบบ **"เลือกโดยผู้ใช้งาน (Dynamic Selection)"**

### การประมวลผลบนระบบหน้าบ้านและหลังบ้าน (UI & API Integration):
1.  **Frontend (UI):** เมื่อ `DIV_Director` เปิดเอกสารขึ้นมา ระบบจะแสดงเมนูตัวเลือกเดียวในการมอบหมายงาน
2.  **Backend (API):** ระบบหลังบ้านจะรันคิวรีตรวจสอบความสัมพันธ์ของพนักงานในองค์กรผ่านตาราง `admin`:
    ```sql
    SELECT a_id, a_name, a_role, a_position 
    FROM   admin 
    WHERE  a_parent_id = :div_director_id
       AND a_role IN ('SEC_DIRECTOR', 'GRP_LEADER');
    ```
    *   *หมายเหตุ:* ข้อมูลในระดับ "ผู้อำนวยการส่วน" และ "หัวหน้ากลุ่มงาน" ทั้งหมดที่รายงานตรงต่อกองนั้นจะแสดงขึ้นมาให้เลือกส่งต่อในแบบฟอร์มเดียวโดยทันที
3.  **State Machine:** เมื่อเลือกบุคคลปลายทางได้แล้ว ระบบจะอัปเดต `in_current_owner_id` ของเอกสารนั้น และบันทึกประวัติการส่งต่อ (`DELEGATED`) ลงตาราง `c_workflow_history` เป็นรายบุคคล

## 4. บทสรุปการประยุกต์ใช้ใน BMA OCSC Circular

จากการวิเคราะห์ระบบ PERN Stack ปัจจุบัน:
*   ระบบมีสิทธิ์การเข้าถึง (RBAC) และการเชื่อมโยงสายผู้บังคับบัญชาผ่าน `a_parent_id` ในโครงสร้างตาราง `admin` อยู่แล้ว
*   ระบบมีฟังก์ชัน `/api/admin/workflow/delegate` และบริการ `WorkflowService` ที่ใช้จัดการสถานะเอกสารและการบันทึกประวัติลงตาราง `c_workflow_history`
*   **ข้อสรุปเชิงสถาปัตยกรรม:** ควรเลือกออกแบบกระบวนการทำงานในลักษณะ **Modular Workflows** ผนวกร่วมกับการใช้ **Dynamic Target Node (ตามแนวทางที่ 2 ของข้อ 3)** เพื่อสร้างสมดุลระหว่างความสะดวกของแอดมินในการสร้าง Template (ไม่ต้องลากเส้นซับซ้อน) ความยืดหยุ่นของผู้อำนวยการกองในชีวิตการทำงานจริง และความเสถียรของโค้ดในระยะยาว (Reflective Thinking)
